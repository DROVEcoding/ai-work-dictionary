# V10B：组织级词库隔离

V10B 的学习主题是：SaaS 的多租户数据隔离。

V10A 让系统有了“组织 / 工作区”。V10B 开始让词条数据真正属于某个组织。

## 这一版做什么

- 新增 `organization_term_backups` 表。
- 每份组织词库都绑定一个 `organization_id`。
- 当前选择哪个组织，上传/下载就操作哪个组织词库。
- 组织成员才能读取和更新这个组织的词库。
- 如果没有选择组织，仍然保留原来的个人云端同步。

## 关键概念

- `organization_id`：组织编号，也可以理解成数据的归属标签。
- `tenant`：租户，通常就是一个客户、团队、公司或工作区。
- `multi-tenant`：多租户，一个系统服务多个客户，但数据必须隔离。
- `RLS`：Row Level Security，数据库按每一行数据判断当前用户能不能访问。

## 为什么重要

真实 SaaS 的核心不是“能登录”，而是：

```text
A 组织的数据，B 组织不能看到。
```

前端隐藏按钮只能改善体验，不能保证安全。真正的隔离要写在数据库 RLS 里。

## 你需要做什么

在 Supabase SQL Editor 执行：

```text
docs/v10b-organization-terms-schema.sql
```

执行成功后：

1. 登录云端账号。
2. 选择一个组织。
3. 修改或新增几个词条。
4. 点击 `上传到云端`，这时上传的是当前组织词库。
5. 切换到另一个组织，词条列表会变成另一个组织的本地缓存。
6. 点击 `从云端恢复`，读取当前组织的云端词库。

## 我做了什么

- 让本地缓存 key 跟随当前组织变化。
- 让上传/下载云端词库时优先使用当前组织。
- 保留个人云端同步作为没有选择组织时的旧模式。
- 新增组织词库 SQL 和 RLS。
- 更新测试，验证组织词库 payload。

## 真实 SaaS 还差什么

V10B 先做“整份词库备份”的组织隔离。更真实的生产系统通常还会继续拆细：

- 每个词条单独一行，而不是整份 JSON 备份；
- 记录每个词条由谁创建、谁最后修改；
- 冲突处理，比如两个人同时编辑同一个词条；
- 邀请成员加入组织；
- 组织 owner / member / viewer 等更细权限。
