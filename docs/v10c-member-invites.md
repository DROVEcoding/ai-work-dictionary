# V10C：组织成员邀请

V10C 的学习主题是：团队协作里的成员关系和 owner 权限。

V10A 有了组织，V10B 让词条属于组织。V10C 开始让组织不再只有一个人。

## 这一版做什么

- 当前组织 owner 可以输入邮箱添加成员。
- 被添加的人必须已经注册云端账号。
- 数据库函数会检查当前用户是不是组织 owner。
- 页面会显示当前组织成员列表。
- 普通 member 可以看到成员列表，但不能添加成员。

## 为什么先做这个轻量版

真实产品常见的邀请流程是：

```text
输入邮箱 -> 发邀请邮件 -> 对方点链接 -> 接受邀请 -> 加入组织
```

这一整套会涉及邮件服务、邀请 token、过期时间、防滥用和通知系统。V10C 先做核心权限模型：

```text
owner 输入已注册用户邮箱 -> 数据库检查 owner 权限 -> 写入 membership
```

这样我们先学最关键的部分：谁有权把谁加入哪个组织。

## 关键概念

- `membership`：成员关系，记录某个用户属于某个组织。
- `owner`：组织拥有者，可以添加成员。
- `member`：组织普通成员，可以使用组织数据，但不能管理成员。
- `RPC / database function`：数据库函数，把多个步骤包成一个安全动作。

## 你需要做什么

在 Supabase SQL Editor 执行：

```text
docs/v10c-member-invites-schema.sql
```

执行成功后：

1. 确认你当前账号是某个组织的 `拥有者`。
2. 让另一个邮箱先注册云端账号。
3. 在当前组织面板输入那个邮箱。
4. 点击 `添加成员`。
5. 成员列表里应该出现这个邮箱，角色为 `成员`。

如果 SQL Editor 返回 `column reference "user_id" is ambiguous`，意思是数据库函数里有多个地方都叫 `user_id`，Postgres 不知道该用哪一个。最新版 SQL 已经用表别名和主键约束消除了这个歧义。

## 真实 SaaS 还差什么

V10C 还不是完整邮件邀请系统。后续可以继续学习：

- 发邮件邀请链接；
- 邀请 token 和过期时间；
- 成员移除；
- 角色升级和降级；
- owner 不能删除最后一个 owner；
- 审计日志，记录谁邀请了谁。
